# 问题定义：从反馈噪声到可执行产品信号

## 1. 背景

智阅云是一款提供OCR、票据识别、表格识别、合同抽取和开放API的文档智能SaaS，覆盖网页、App和微信小程序。产品上线后，应用市场、客服工单和销售登记持续产生大量反馈，但这些反馈格式不同、质量不一、重复严重，产品经理无法稳定回答“用户最痛的三件事是什么、哪个应该先改、下一步由谁处理”。

本项目当前限定CSV输入，覆盖应用市场评论、客服问题反馈、销售需求登记、角色表、需求池和人工标注集，不处理聊天记录及邮件TXT。

## 2. 核心问题陈述

> 产品、客服和需求承接团队缺少一套可解释的反馈分诊机制，无法从高噪声、多表达、多渠道的CSV反馈中及时识别共性问题、评估业务影响、关联已有需求并转化为行动，导致决策依赖人工阅读、最近记忆和个人判断。

问题不是“没有用户反馈”，而是反馈无法被可靠地压缩成产品决策。

## 3. 目标用户与待完成任务

| 用户 | 核心任务 | 当前阻碍 | 期望结果 |
|---|---|---|---|
| 产品经理 | 找到Top痛点并排优先级 | 反馈分散、重复、缺少证据 | 5分钟看懂问题、影响和建议行动 |
| 客服团队 | 快速、准确地回应用户 | 咨询重复、事实口径不一致 | 获得知识库约束的回复草稿 |
| 需求承接团队 | 把高价值反馈录入需求池 | 新旧需求难查重、背景不完整 | 关联已有需求或生成结构化候选需求 |
| 研发/算法团队 | 定位真正高影响的问题 | 缺少规模、复现和业务严重度 | 收到带证据、状态和临时方案的行动卡片 |

## 4. 典型症状

- 同一问题有几十种表述，如“章压数字”“价税合计错”“8600识别成3600”。
- 星级与文本情感冲突，“五星”可能是对广告弹窗的反讽。
- 销售登记中混有已上线能力、Bug、重复需求和真正的新需求。
- 一条反馈可能同时包含咨询、问题和需求，直接按整行分类会丢失信息。
- VIP续约风险、财务金额错误和普通体验吐槽混在一起，纯频次排序会失真。
- 用户群或历史备注中的事实可能过时，自动回复存在错误承诺风险。

## 5. 根因

1. **数据层**：CSV字段不统一、存在错列、缺失、未转义逗号和重复文本。
2. **语义层**：反馈具有反讽、错字、中英混合、业务缩写和同义表达。
3. **决策层**：缺少统一的严重度、影响面、客户价值和趋势权重。
4. **流程层**：洞察、客服回复、Bug工单与需求池之间没有闭环。
5. **治理层**：缺少置信度、证据追溯和人工审核边界。

## 6. 解决范围

### 本期范围

- 本地CSV导入、字段修复与统一数据模型；
- 噪声、情感、主题和反馈类型识别；
- 共性问题聚类和代表证据提取；
- 热点排序、需求池查重、回复草稿与行动卡片；
- 一页洞察看板和独立评估页面；
- 高风险或低置信度结果的人工确认。

### 非目标

- 不自动发送公开回复；
- 不自动修改正式需求池或研发工单；
- 不承诺完全替代产品经理判断；
- 不把星级直接等同于文本情感；
- 当前不抓取实时外部平台数据，不处理TXT聊天和邮件。

## 7. 成功指标

| 维度 | 指标 | 当前基线/目标 |
|---|---|---|
| 数据处理 | CSV成功解析率 | 支持6类CSV，异常列可修复 |
| 情感质量 | 情感准确率 / Macro-F1 | 当前87.3% / 约87% |
| 主题质量 | 主题准确率 / Macro-F1 | 当前74.0% / 约69% |
| 洞察效率 | PM理解Top问题时间 | 目标不超过5分钟 |
| 可追溯性 | Top问题代表证据覆盖率 | 每个主题至少3条代表反馈 |
| 行动转化 | 高价值反馈分流覆盖率 | 可进入Bug、已有需求或候选需求 |
| 安全治理 | 高风险自动执行率 | 0%，必须人工确认 |

## 8. 验收问题

系统上线或演示时必须能够回答：

1. 当前最值得优先处理的三个问题是什么？
2. 每个结论有多少条反馈、多少独立用户、哪些渠道和哪些代表原文？
3. 排序为何如此，是否考虑了财务风险、VIP续约和时间趋势？
4. 该问题应回复、建Bug、关联已有需求，还是创建候选需求？
5. 模型在哪里可能出错，哪些结果必须由人确认？
